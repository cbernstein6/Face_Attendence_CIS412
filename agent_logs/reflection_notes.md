During our work on the Face recognition attendance system, we used Gemini CLI and better learned how to use it. Tasks that were most suitable for Gemini was using it during the concept stage for the project as it was very good at understanding what was required of the program as well as possible ways to achieve that goal. Additionally it was very good at debugging code and providing possible solutions when they arose. another suitable task for Gemini was to make it generate Code for a single module, such as for the enroll.py script. We left all of our small coding tasks to Gemini, and they were all handled very effectively. Also, the creation of a venv was handled by gemini, which made setting up the project and it's dependencies very easy, and allowed the project to start smoothly.

However, it required a lot of fine tuning and for you to specify specifically what you wanted the CLI to do, otherwise it would produce a lot of variables and code that was useless and not relevant to the projects specifications. Gemini produces a lot of AI hallucinations in its code so it would have to be fine tuned after generation by human hands, making it not suitable for solo coding. Organization was also not suitable for Gemini because it would produce a lot of nonsense unrelated to the project and doesn’t understand the full scope of the project. Because of this, tasks like design and planning were all done by humans. We also found that code refinement was much easier manually, as gemini seemed to prefer adding more code rather than slim it down when an issue arose.

Prompts started off somewhat general, but after continuous attempts they were made more detailed for fine-tuned code. For instance, for the main file, I started off by telling it to write a python file to tie my individual modules together. However, after the initial results weren't what I was hoping for I made the prompt much more specific to exactly what I was looking for in the file. After the main file was a success with the more specific prompting, I used the same pattern to develop the other modules, which was a much more challenging way of using gemini.

Using an ai agent changed my approach to software development to something of a manager for the dev. Instead of focusing on the lower level files, I first figured out the design I wanted and broke it down into manageable chunks for the ai to handle. Thus, the focus was much more on the system rather than individual files (similar to a manager's role).

Our team found that over-reliance is the biggest challenge when using AI Agents, as they are very good at writing code when prompted correctly. This is such an important issue since even if the code works perfectly, if the dev doesn't understand all the code and why this solution is preferred over alternatives, they shouldn't be merging the code. Also, they aren't even providing much to the team if they are relying on ai for everything.
